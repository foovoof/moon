# سياسة الأمان — Security Policy

## الإصدارات المدعومة

| الإصدار | مدعوم |
|---------|--------|
| `main` (تطوير نشط) | ✅ |
| إصدارات موسومة أقدم من آخر إصدارين | ❌ |

## الإبلاغ عن ثغرة أمنية

**لا تفتح Issue أو Discussion علنية لأي ثغرة أمنية.**

الرجاء اتباع الخطوات التالية:

1. راسل فريق الأمان عبر القناة الخاصة الموضحة في `governance/roles/admin.md`
   و `SECURITY.md` (البريد/النموذج الرسمي يُحدَّد قبل أول إصدار عام).
2. قدّم أكبر قدر ممكن من التفاصيل:
   - وصف الثغرة وتأثيرها المحتمل.
   - خطوات إعادة الإنتاج (Proof of Concept إن وُجد).
   - النسخة/الـ commit المتأثر.
3. سيتم الرد خلال **72 ساعة عمل** كحد أقصى بإشعار استلام.
4. سيتم إصلاح الثغرات الحرجة على أولوية قصوى، مع نشر إفصاح منسّق
   (Coordinated Disclosure) بعد توفر الإصلاح.

## نطاقات حساسة خاصة بهذا المشروع

بحكم طبيعة **Moon** كمنصة استخباراتية/تحقيقية، تُعامل الثغرات التالية
كحرجة (Critical) بشكل تلقائي:

- أي تسرب أو تجاوز صلاحيات يسمح بالوصول لبيانات أدلة (`domain/evidence.py`)
  أو سجلات تحقيقات (`domain/investigation.py`) دون تفويض.
- أي كسر في `security/authorization/` أو `policy_engine/authorization/`
  يسمح بتجاوز الأدوار المعرّفة في `governance/roles/`.
- أي تسريب لأسرار/مفاتيح API عبر `security/secrets/` أو ملفات `.env`.
- أي إخلال بسلسلة حفظ الأدلة (`domain/chain_of_custody.py`).

## ممارسات الأمان في الكود

- لا تضع مفاتيح API أو أسرار في الكود المصدري — استخدم `.env` (غير مُتتبَّع)
  و `security/secrets/provider.py`.
- جميع نقاط الدخول في `apps/api/` يجب أن تمر عبر `apps/api/middleware.py`
  للمصادقة والتفويض.
- راجع `governance/policies/privacy_policy.yaml` قبل إضافة أي مُجمّع بيانات
  جديد في `collectors/`.
- استخدم `pre-commit` (`.pre-commit-config.yaml`) لمنع تسريب الأسرار عبر
  `detect-secrets` قبل كل commit.

## الاعتماديات

يتم فحص الاعتماديات دوريًا عبر `dependabot` (`.github/dependabot.yml`)
لكل من `pyproject.toml` و `package.json`.
